## 1. 项目概述

### 1.1 项目定位

**数乡计划**是一个服务大学生「三下乡」社会实践的配套平台，核心目标是沉淀出一个可长期运营的**乡村数字资源库**——不是一次性的活动网站，而是一个会持续生长的乡村数据平台。

### 1.2 用户角色

| 角色 | 核心诉求 |
|------|---------|
| **实践队（核心用户）** | 策划有方向、执行不遗漏、成果拿得出手 |
| **村庄 / 村干部** | 让实践带来的改变被看见、留得下 |
| **后续队 / 评审老师 / 资助方** | 承接经验、对比成效、验证价值 |

### 1.3 核心理念

- **从静态展示到智能平台**：把网站的六大栏目内容作为 AI 的知识库，让平台能检索、能引用、能生成
- **全程陪伴**：一份贯穿实践始末的档案，策划 → 执行 → 成果三阶段数据不割裂
- **生成式可视化 + DIY**：AI 智能编排打底 + 队员低代码微调，破解「千队一面」与「门槛太高」

## 2. 从「展示型网站」升级为「功能型平台」

### 2.0 问题诊断：现状 vs 愿景

当前平台六个栏目中，**五个是只读陈列，只有一个子功能（我的实践）有交互闭环**：

| 栏目 | 现状 | 用户能做什么 |
|------|------|-------------|
| 乡村百科 | 12 个村档案 + 地图下钻 | 只能看，不能贡献 |
| 乡村实践 | 成果长廊浏览 | 只能看别人的成果 |
| **→ 我的实践** | **AI 三阶段工作台** | **★ 唯一的功能闭环** |
| 乡村之声 | 需求列表展示 | 只能看需求，不能真正对接 |
| 实践攻略 | 静态攻略文章 | 只能看模板，不能个性化 |
| 乡村好物 | 商品卡片展示 | 只能看商品，不能下单/联系 |
| 关于我们 | 团队介绍 | 只能看 |

**核心矛盾**：平台定位是「会生长的乡村数字资源库」，但生长靠的是用户持续贡献数据——而现有平台除了「我的实践」，没有任何让用户主动贡献数据的入口。一个只有消费没有生产的平台，不会生长。

### 2.1 功能化改造总览

下面按栏目逐一给出功能化改造方案。每个方案的衡量标准是：**用户离开时，平台上多了一些只有 ta 才能留下的东西。**

```
                    ┌── 消费型（只读）──→ 功能型（可交互）──┐
                    │                                      │
乡村百科  ───────── 看档案  ──→  贡献村料、对比村庄、数据纠错  │
乡村之声  ───────── 看需求  ──→  揭榜响应、发布需求、追踪进度  │
乡村好物  ───────── 看商品  ──→  下单购买、上架好物、评价体系  │
实践攻略  ───────── 看攻略  ──→  个性化清单、模板定制、经验贡献  │
乡村实践  ───────── 看成果  ──→  我的实践（已有）+ 成果评测    │
关于我们  ───────── 看介绍  ──→  社区贡献榜、数据贡献者激励      │
                    │                                      │
                    └── 每个栏目从「我来看看」→「我来用、我来贡献」┘
```

### 2.2 乡村百科 —— 从「少数人编纂」到「人人共建」

**现状**：12 个名村的档案由开发团队手工录入 JSON 文件，用户只能浏览检索。

**功能化改造**：

| 功能 | 说明 | 优先级 |
|------|------|--------|
| **村民/队员补充入口** | 每个村庄详情页设「补充信息」按钮，用户可提交文字、照片、数据指标（如人口、收入），经审核后并入村庄档案 | ★★★ |
| **村庄对比器** | 选 2-4 个村，并排对比人口、产业、荣誉、资源多维度雷达图，帮助实践队选题时快速评估 | ★★★ |
| **数据纠错机制** | 用户发现档案数据过时或错误，可提交纠错建议（附来源），审核通过后档案更新 + 贡献者留名 | ★★☆ |
| **村庄动态时间线** | 不只是静态快照——展示村庄数据随时间的变化曲线（历年人口、收入、产业变迁），让档案「长」起来 | ★★☆ |
| **「我来过这个村」标记** | 实践队到达某村后可标记，形成「此村接待过哪些队伍」的关联链，后续队伍参考前人足迹 | ★★☆ |

**数据贡献闭环**：用户贡献 → 审核通过 → 档案生长 → 贡献者获得声望徽章 → 激励更多贡献。这才是「生长」的引擎。

### 2.3 乡村之声 —— 从「需求公示栏」到「双向对接引擎」

**现状**：展示乡村需求列表，但没有响应/对接的工作流。

**功能化改造**：

| 功能 | 说明 | 优先级 |
|------|------|--------|
| **揭榜响应系统** | 实践队看到需求后可点击「响应此需求」，提交自己的方案构想，需求发布方（乡镇/村）查看并确认合作 | ★★★ |
| **需求发布功能** | 不只是管理员录入——村干/乡镇经认证后可自行发布需求（包含预算、时限、期望成果），形成真实的需求供给闭环 | ★★★ |
| **对接进度追踪** | 需求从「待响应」→「已对接」→「进行中」→「已完成」的状态流转，双方可更新进展、上传阶段性成果 | ★★☆ |
| **需求匹配推荐** | AI 根据实践队的专业、往期成果，主动推荐适合的需求（「这 3 条电商帮扶需求可能适合你们队」） | ★★☆ |
| **评价体系** | 需求方对完成质量打分，评价沉淀为实践队的信用资产，后续对接时可展示 | ★☆☆ |

**从展示到对接的关键**：需求有状态机、双方有互动通道、结果有评价反馈。这才是「乡村出题，青年回应」的真正落地。

### 2.4 乡村好物 —— 从「商品橱窗」到「消费帮扶通道」

**现状**：商品卡片展示，没有购买/下单能力。

**功能化改造**：

| 功能 | 说明 | 优先级 |
|------|------|--------|
| **购买意向 / 联系农户** | 点击「想买」后展示农户联系方式或跳转电商链接（避免自建支付体系的高门槛），至少打通「看到→想买→联系上」 | ★★★ |
| **好物上架申请** | 村民/实践队可提交新好物（产品图、描述、价格、产量），审核通过后上线 | ★★☆ |
| **好物故事页** | 每件好物绑定背后的村庄与人（「这件竹编出自小朱湾村陈大爷之手，手艺传承三代」），消费也是帮扶叙事 | ★★☆ |
| **收藏与分享** | 收藏心仪好物、生成分享卡片（朋友圈/公众号），用传播带动消费 | ★★☆ |

**核心思路**：不自建电商重资产，而是做「发现层」+「连接层」——让好物被看见、故事被传播、买家与农户能对接。真正的交易走外部电商链接或线下联系。

### 2.5 实践攻略 —— 从「静态文章」到「个性化工具箱」

**现状**：攻略以静态 JSON 数据渲染，所有人看到的都一样。

**功能化改造**：

| 功能 | 说明 | 优先级 |
|------|------|--------|
| **智能准备清单** | 根据队伍选题（电商帮扶 / 文化调研 / 支教…），AI 生成专属的行前准备清单（工具、模板、采集指标），而非千人一面 | ★★★ |
| **模板定制下载** | 调研问卷模板、访谈提纲模板——AI 根据目标村和选题自动调整字段，队员可在线编辑后导出 Word/Excel | ★★★ |
| **攻略贡献入口** | 完成实践后，队员可提交「我们踩过的坑」「我们觉得最有效的调研方法」，经审核汇入攻略库 | ★★☆ |
| **互动式安全检查** | 行前安全自检清单（包含目的地天气、地质灾害预警等实时信息），逐项打勾确认 | ★☆☆ |

### 2.6 乡村实践（成果长廊）—— 从「作品集」到「成果评测 + 经验复用」

**现状**：「我的实践」子功能已有完整三阶段闭环；但公共成果长廊仍是静态展示。

**功能化改造**：

| 功能 | 说明 | 优先级 |
|------|------|--------|
| **成果评分与反馈** | 评审老师/其他队员可对公开成果点赞、评论、打分（维度：数据完整度 / 可视化质量 / 可复制性），帮助优质成果浮出 | ★★☆ |
| **成果复用** | 「用这个模板创建我的实践」——看到一份优秀的成果页，一键复制其可视化布局和指标体系到自己的实践档案中，填入自己的数据 | ★★★ |
| **成果对比** | 并排查看多支队伍在同一村庄/同一选题下的成果，直观对比不同方法的成效差异 | ★★☆ |
| **年度报告** | 每年自动生成「三下乡年度实践报告」：覆盖多少村、多少队伍、产出多少成果、数据趋势——平台自身也成为被研究对象 | ★☆☆ |

### 2.7 跨栏目的功能性基础设施

以下功能不属于单一栏目，而是服务于「从展示到功能」的底层能力：

| 基础设施 | 说明 |
|----------|------|
| **贡献者体系** | 统一的积分/徽章/声望：贡献村庄数据、响应需求、上架好物、发布攻略——所有贡献行为统一计分，个人主页展示贡献足迹 |
| **通知系统** | 需求被响应了、补充的数据审核通过了、成果收到了新评论——让用户有回来的理由 |
| **个人主页** | 我的实践档案 + 我的贡献记录 + 我的声望徽章 + 我收藏的村庄/好物，聚合一个人的所有痕迹 |
| **搜索能力升级** | 当前只有首页的地名搜索。升级为全站统一搜索：搜一个村名 → 返回该村的百科档案、相关实践成果、相关需求、相关好物——一站式信息聚合 |
| **数据 API** | 开放村庄基础数据 API（人口、产业、荣誉等），第三方可调用做研究或二次开发，拓展平台影响力 |

### 2.8 功能化优先级路线图

```
第一阶段（实训期 · 当前）
├─ ★ 我的实践 AI 智能体（三阶段闭环）       ← 已有，持续打磨
├─ 村庄百科「补充信息」入口                  ← 数据贡献的第一个入口
├─ 乡村之声「揭榜响应」                      ← 需求对接的最小闭环
└─ 乡村好物「购买意向 / 联系农户」            ← 消费帮扶的最小闭环

第二阶段（比赛期）
├─ 村庄对比器                                ← 帮助选题决策
├─ 成果复用（一键复制模板）                   ← 降低新队伍上手成本
├─ 智能准备清单 + 模板定制下载                ← 攻略从静态变动态
├─ 通知系统                                  ← 让用户有理由回来
└─ 贡献者体系                                ← 平台生长的引擎启动

第三阶段（长期运营）
├─ 需求匹配推荐（AI 推荐适合队伍的需求）
├─ 数据 API 开放
├─ 年度报告自动生成
└─ 社区评价体系（打分/评论/排行）
```

### 2.9 衡量标准

改造是否有效，不看功能列表有多长，看这几个指标：

| 指标 | 当前 | 目标 |
|------|------|------|
| **用户贡献数据量** | 0（只有开发团队录入） | 每个队伍至少贡献 1 条新数据（村庄信息/需求/好物/攻略） |
| **回访率** | 无数据（没有通知、没有个人页） | 有通知触发后的回访行为 |
| **跨栏目联动** | 无（各栏目互相不知道对方存在） | 从任意入口都能关联到其他栏目的相关内容 |
| **平台数据增量** | 12 个村档案（静态） | 档案数持续增长，且增量来自用户而非开发团队 |


### 2.2 架构原则

1. **前后端分离**：Vue 3 SPA + Express REST API，开发期 Vite 代理转发，生产期 Express 托管 dist/
2. **模块化前端**：每个一级栏目 = 一个独立模块，新增栏目不改老代码
3. **依赖注入后端**：Express app 通过工厂函数 `createApp({ db, secret })` 组装，便于测试
4. **AI 能力服务端封装**：API Key 仅存服务端，前端不感知模型细节

### 2.3 技术分层

| 层 | 职责 | 技术 |
|----|------|------|
| 展示层 | UI 渲染、图表可视化、交互 | Vue 3 + ECharts + 纯 CSS |
| 路由层 | SPA 路由、模块路由自动发现 | Vue Router (Hash) + `import.meta.glob` |
| 状态层 | 组件内响应式 + 组合式 API | Vue 3 Composition API |
| 服务层 | 业务逻辑、AI 编排、数据校验 | Express Services |
| 数据层 | 持久化、查询、迁移 | SQLite + 手动迁移脚本 |
| AI 层 | 方案生成、联网搜索、意图编排 | DeepSeek API + 博查 AI Search |

## 3. 技术选型

### 3.1 前端

| 技术 | 版本 | 用途 | 选型理由 |

| **@vitejs/plugin-vue** | ^5.2.1 | Vite Vue 插件 | SFC 编译支持 |

### 3.2 后端

| 技术 | 版本 | 用途 | 选型理由 |
|------|------|------|---------|
| **Express** | ^5.2.1 | HTTP 框架 | 生态最大，中间件丰富 |
| **better-sqlite3** | ^12.11.1 | 数据库 | 同步 API、零配置、单文件部署 |
| **jsonwebtoken** | ^9.0.3 | JWT 认证 | 无状态鉴权，适合 SPA |
| **bcryptjs** | ^3.0.3 | 密码哈希 | 纯 JS 实现，跨平台免编译 |

### 3.3 AI 能力

| 项目 | 说明 |

### 5.2 依赖注入模式

`server/app.js` 通过工厂函数组装 Express 应用，所有路由通过函数参数接收依赖：

```javascript
// server/app.js
export function createApp({ db, secret }) {
  const app = express()
  app.use('/api/auth',    makeAuthRouter(db, secret))
  app.use('/api/teams',   makeTeamsRouter(db, secret))
  app.use('/api/dossiers', makeDossiersRouter(db, secret))
  app.use('/api/plan',    makePlanRouter(db, secret))
  app.use('/api/search',  makeSearchRouter(db, secret))

}
```

## 6. 数据库设计

### 6.1 ER 图

```
┌──────────┐       ┌──────────────┐       ┌──────────┐
│  users   │       │ memberships  │       │  teams   │
├──────────┤       ├──────────────┤       ├──────────┤
│ id (PK)  │──←──→│ user_id (FK) │──←──→│ id (PK)  │
│ username │       │ team_id (FK) │       │ name     │
│ password │       │ role         │       │invite_code│
│display_n.│       │ joined_at    │       │created_by│
│created_at│       └──────────────┘       │created_at│
└──────────┘                              └──────────┘
     │                                          │
     │ created_by                               │ team_id
     ↓                                          ↓
┌──────────────────────────────────────────────────────┐
│                      dossiers                         │
├──────────────────────────────────────────────────────┤
│ id (PK, TEXT)     │ title    │ stage                  │
│ team_id (FK)      │ created_by (FK)                   │
│ payload (JSON)    │ created_at │ updated_at           │
└──────────────────────────────────────────────────────┘
```

### 6.2 表结构说明

| 表 | 说明 | 关键字段 |
|----|------|---------|
| **users** | 用户账号 | `username`（唯一）、`password`（bcrypt 哈希） |
| **teams** | 实践队伍 | `invite_code`（唯一，随机生成） |
| **memberships** | 用户↔队伍 多对多 | `role`：`owner`（建队人）\| `member` |
| **dossiers** | 实践档案 | `stage`：`plan`\|`track`\|`result`；`payload` 存完整 JSON |

**设计决策**：
- 档案主体 `payload` 采用 JSON 文本整存，而非按字段拆列——因为方案结构随迭代频繁变化，JSON 列灵活性远优于频繁 DDL 迁移
- `memberships` 独立表支持用户同时属于多个队伍
- 复合索引 `(team_id, updated_at)` 优化列表页最常见的查询模式

### 6.3 迁移策略

- DDL 写在 `db/schema.sql`，全程使用 `CREATE ... IF NOT EXISTS`
- `db/migrate.js` 在服务启动时执行，幂等可重复跑
- 暂无不兼容的 schema 变更，后续若需要，采用增量迁移脚本

### 7.3 AI 调用流程

```
用户输入 "去小朱湾村做电商帮扶"
  → POST /api/plan/generate  { village, idea }
    → planService.generate()
      1. searchService.searchVillage()   → 博查联网搜索
      2. 匹配平台内已有数据（百科档案、往届成果）
      3. deepseek.chatJSON()             → AI 编排方案
```

## 8. 核心功能：AI 实践智能体

### 8.1 三阶段流水线

```
┌─────────────────┐     ┌─────────────────┐     ┌─────────────────┐
│  阶段一 · 策划   │ ──→ │  阶段二 · 执行   │ ──→ │  阶段三 · 成果   │
│  Plan            │     │  Track           │     │  Result          │
├─────────────────┤     ├─────────────────┤     ├─────────────────┤
│ · AI 检索全站    │     │ · 材料自动归档   │     │ · AI 智能编排     │
│ · 联网搜索补充   │     │ · 对照方案查缺   │     │ · 可视化组件拼装  │
│ · 生成方案初稿   │     │ · 实时进度提醒   │     │ · DIY 搭建台     │
│ · 多轮对话丰富   │     │ · 漏采预警       │     │ · 一键导出       │
└─────────────────┘     └─────────────────┘     └─────────────────┘
        ↓                       ↓                       ↓
   方案 JSON              采集材料归档             可视化成品
   (schema 约束)          (对照方案校验)           (组件化拼装)
```

### 8.2 阶段一：策划期

**输入**：队员手填 idea + 可选上传文档/图片
**处理**：
1. 解析 idea 关键词，在平台六大栏目内检索（百科档案、往届成果、攻略模板、乡村需求）
2. 平台无匹配时，调用博查联网搜索补充
3. 将检索结果灌入 DeepSeek，按 `planSchema` 生成结构化方案

**输出（方案 JSON Schema）**：
```json
{
  "title": "实践标题",
  "village": "目标村",
  "topic": "选题方向",
  "indicators": [
    { "name": "指标名", "unit": "单位", "baseline": "采集方法" }
  ],
  "timeline": [
    { "phase": "阶段", "tasks": ["任务"], "duration": "预计天数" }
  ],
  "expectedOutcomes": ["预期成果"],
  "references": [
    { "type": "platform|web", "title": "来源标题", "url": "跳转链接" }
  ]
}
```

### 8.3 阶段二：执行期

**输入**：队员持续上传的实地数据（照片、表格、访谈记录）
**处理**：
1. 材料自动归入档案
2. 调用 `gapAnalysis.js` 对照方案指标逐项检查覆盖率
3. 生成缺漏提醒和建议

**核心逻辑**：
```javascript
// gapAnalysis 核心（简化）
function analyzeGap(plan, collected) {
  return plan.indicators
    .filter(ind => !collected.some(c => c.name === ind.name))
    .map(ind => ({ indicator: ind.name, suggestion: `建议采集：${ind.baseline}` }))
}
```

### 8.4 阶段三：成果期（★ 重点）

**输入**：阶段一方案 + 阶段二采集材料
**处理**：
1. AI 从原始材料中提取结构化数据（指标数值、人物信息、时间事件）
2. 按叙事意图分类编排（讲变化 / 讲过程 / 讲人物 / 讲分布）
3. 从可视化组件库挑选匹配组件，拼装成果页
4. 队员可在搭建台拖拽调整、替换组件、补充文案

**成果输出方向**（8 个成品方向）：
1. AI 村庄导览图（交互地图 + 实践足迹标注）
2. AI 成果海报（标题 + 核心数据 + 亮点图）
3. 帮扶前后对比图（指标 A/B 对比可视化）
4. 实践足迹时间线（按时间串起事件）
5. 人物故事墙（图文卡片墙）
6. 成果数据看板（仪表盘指标汇总）
7. 一村一实践长图（整合导出）
8. 宣传文案 / 短视频脚本

## 10. 安全与认证

### 10.1 认证流程

```
注册：POST /api/auth/register { username, password }
  → bcrypt 哈希密码 → 写入 users 表 → 返回 { id, username }

登录：POST /api/auth/login { username, password }
  → 查 users 表 → bcrypt 比对 → 签发 JWT（7 天有效期）
  → 返回 { token, user: { id, username } }

后续请求：
  Authorization: Bearer <token>
  → auth 中间件解析 JWT → 挂载 req.user → 业务路由
```

### 10.2 安全措施

| 措施 | 实现 |
|------|------|
| 密码存储 | bcrypt 哈希（10 轮 salt），绝不存明文 |
| JWT 密钥 | 服务端 `.env` 配置，不入仓；生产环境使用长随机串 |
| 请求体限制 | `express.json({ limit: '1mb' })` 防大 payload |
| API Key 隔离 | DeepSeek/博查密钥仅存服务端 `.env`，前端不可见 |
| 档案归属校验 | 每个 dossier 操作前校验 `team_id` 归属 |
| 邀请码 | 8 位随机字符串，加入队伍时必需 |

### 10.3 权限矩阵

| 操作 | owner | member | 未登录 |

## 12. 工程规范

### 12.1 开发流程

```
需求分析 → 设计文档（spec）→ 实现计划（plan）→ 编码 → 测试 → 代码审查 → 合并
```

- **设计先行**：每个 feature 先写 spec 文档（`docs/superpowers/specs/`）明确接口、数据结构、交互流程

- **测试驱动**：核心逻辑先写测试用例，保证基线不回归

### 12.2 代码规范

- **提交信息**：`type: description` 格式（`feat:` / `fix:` / `refactor:` / `test:` / `docs:`）]，末尾附带 `Co-Authored-By: Claude Fable 5 <noreply@anthropic.com>`
- **文件命名**：kebab-case（`village-info-card.vue`），模块入口统一 `IndexView.vue`
- **组件风格**：Vue 3 Composition API（`<script setup>`），组合式函数抽离复用逻辑
- **CSS**：全局主题变量（`theme.css`），组件内 scoped 样式

### 12.3 分支策略

- `main` 分支：可演示的功能主线
- Feature 开发：在 `main` 上直接迭代（小团队模式），重大重构使用 worktree 隔离
