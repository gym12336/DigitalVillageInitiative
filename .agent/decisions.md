# 架构决策记录（ADR）

> 本文档记录数乡平台的关键技术决策及其理由，供 AI 与团队理解「为什么这样做」。
> 每条决策格式：背景 → 决策 → 理由 → 影响。

---

## ADR-001：AI 禁飞区制度

- **背景**：实训要求核心功能必须人类掌握，不能当 AI 的「搬运工」。
- **决策**：认证授权、AI 输出校验（planSchema）、LLM 安全通道（deepseek）三块**必须手写**，禁止 AI 生成。
- **理由**：这三块是安全边界与信任边界，必须由人完全掌控、能逐行讲解。
- **影响**：详见 [no-fly-zone.md](./no-fly-zone.md)。AI 读到这些文件时只可参考，不得改写。

---

## ADR-002：数据库选用 SQLite（better-sqlite3）

- **背景**：需要一个后端持久化方案，团队 2 人、部署环境简单。
- **决策**：用 `better-sqlite3` 单文件数据库，**同步 API**。
- **理由**：单文件、零配置、易部署、易备份；同步 API 代码更直观，适合轻量场景。
- **影响**：曾评估 PostgreSQL，因部署/运维成本过高**否决**。数据量增大后可再迁移。

## ADR-003：认证用自研 JWT + bcrypt，不引入认证框架

- **背景**：需要登录鉴权，但也是禁飞区①（必须手写、可讲解）。
- **决策**：只用 `jsonwebtoken` + `bcryptjs` 两个底层库，鉴权逻辑全部自己写。
- **理由**：框架封装太多黑盒，讲不清；自己写 114 行反而每一步都能解释。
- **影响**：token 载荷只存 userId；验签后再查库防「用户已删」；登录统一 401 防用户名枚举。

## ADR-004：AI「辅助但不依赖」——双引擎兜底

- **背景**：LLM 可能超时、无 key、返回幻觉，系统不能因此瘫痪。
- **决策**：AI 生成的方案必须过 `validatePlanShape` 校验（禁飞区②）；不合格或 LLM 不可用时，走本地规则引擎 `planTemplate.js` 兜底。
- **理由**：AI 输出不可信，必须有防火墙 + 离线兜底，保证系统永远能出方案。
- **影响**：方案对象带 `source: 'ai' | 'template'` 字段，可追溯来源。

## ADR-005：后端依赖注入（无 DI 容器）

- **背景**：需要让 db、secret 等依赖可测、可替换。
- **决策**：所有路由/中间件工厂函数签名统一为 `(db, secret) => Router/中间件`，手动传参组装。
- **理由**：简单透明、无「魔法」，单元测试能塞内存库/假 secret。
- **影响**：`createApp({ db, secret })` 统一组装；禁飞区代码全部可脱离真实环境单测。

---

## ADR-006：前端模块自动路由收集

- **背景**：功能模块多，手动注册路由易漏、易冲突。
- **决策**：`src/router/index.js` 用 `import.meta.glob` 自动扫描所有 `modules/*/routes.js`。
- **理由**：新增模块零侵入，不改动路由入口文件。
- **影响**：新增模块三步走（复制模板 → 加配置 → 定义路由）即自动生效。

---

## ADR-007：LLM 调用统一走 deepseek.js 单一入口

- **背景**：API Key 泄露风险 + 调用需统一超时/重试。
- **决策**：所有 LLM 调用只经 `server/lib/deepseek.js`（禁飞区③），密钥只在服务端读。
- **理由**：密钥隔离、错误分类、20s 超时、单次重试统一治理，前端永远拿不到 key。
- **影响**：见 [no-fly-zone.md](./no-fly-zone.md) 禁飞区③。
