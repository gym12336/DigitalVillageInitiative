# 开发工作流

> Spec-Driven Development：设计先行、计划驱动、逐步验证。

---

## 标准开发流程

```
需求输入 → 设计文档 → 实施计划 → 分步实现 → 验证测试 → 提交
```

### 1. 设计文档 (Spec)

位置：`docs/superpowers/specs/<date>-<feature>-design.md`

内容必须包含：
- **需求背景**：为什么做、解决什么问题
- **功能描述**：做什么、用户能完成什么
- **UI 交互**：界面布局、操作流程（文字或 ASCII 线框图）
- **数据结构**：涉及的数据库表 / JSON 结构 / API 契约
- **边界条件**：空态、错误态、极端情况

### 2. 实施计划 (Plan)

位置：`docs/superpowers/plans/<date>-<feature>.md`

计划内容：
- 任务拆分（每一步可独立验证）
- 涉及的文件清单
- 任务间的依赖关系
- 预计改动范围（大 / 中 / 小）

### 3. 分步实现

按照 plan 的顺序逐任务执行：
1. 读相关代码，理解上下文
2. 写代码（匹配现有模式，不另起炉灶）
3. 完成后标记该任务 done

### 4. 验证

至少做以下验证：
- 前端：`npm run dev`，在浏览器中确认功能正常
- 后端：`npm test`，所有测试通过
- 手工验证关键路径和边界条件

### 5. 提交

```bash
git add <changed-files>
git commit -m "feat: <描述>"
```

---

## AI 协作原则

### 设计阶段
- **先读后写**：实现前必须先读现有代码，理解上下文和模式
- **最小改动**：优先在现有框架内扩展，不做重构式改动
- **匹配风格**：新代码应与周围代码风格一致（命名、注释、结构）

### 实现阶段
- **一次一个任务**：按 plan 顺序逐任务执行，完成一个再下一个
- **原子提交**：每个提交是一个可独立回滚的变更单元
- **零破坏**：新增模块不改动老代码（路由自动收集、模板复制等机制保证）

### 验证阶段
- **写完后立即验证**：不攒到最后一个大 PR
- **测试覆盖关键路径**：不要求 100% 覆盖率，但主流程不能缺

## 实施计划模板

```markdown
# <功能名称> 实施计划

> 基于设计文档：`docs/superpowers/specs/<date>-<feature>-design.md`

## 任务列表

- [ ] 1. 数据库迁移（如有新表）
- [ ] 2. 后端 Service 和 Route
- [ ] 3. 前端 API 封装
- [ ] 4. 前端页面和组件
- [ ] 5. 路由注册
- [ ] 6. 验证测试

## 涉及文件

| 文件 | 改动类型 | 说明 |
|------|---------|------|
| `server/db/migrate.js` | 修改 | 新增 xxx 表 |
| `server/services/xxxService.js` | 新增 | ... |
| `src/modules/xxx/IndexView.vue` | 新增 | ... |
```
