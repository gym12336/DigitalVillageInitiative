## 0. 一句话定位

> 数乡计划不是做一个展示型的内容网站,而是把 AI 做进平台的**采集与知识化命脉**——用 AI 完成实践素材的结构化入库,并基于资源库做 RAG 知识问答。**AI 贯穿核心业务,而非附加聊天框。**

这句话直接回应实训要求「所有项目 AI 贯穿核心业务流程,禁止『主业务+附加聊天框』的浅层集成」「AI 深度融入核心业务,淘汰低价值 CRUD」。

## 1. 背景与选题理由

### 项目现状(半成品基础)

数乡计划是大学生社会实践的落地平台:50 名队员分队赴湖北各村(样板村武汉江夏小朱湾村)实地采集村庄资源、人物故事、影像、新闻稿,持续沉淀到一个可长期运营的乡村数字资源库。

目前已有的半成品:

- Vue3 + Vite 重构的前端框架,暖绿主题,7 大栏目(乡村百科 / 乡村实践 / 乡村之声 / 实践攻略 / 乡村好物 / 关于我们 + 首页)。
- 乡村百科、乡村实践、乡村之声已用真实数据填充;首页 2D 全国地图下钻(全国→省→市→区县)。
- 工程基础:spec/plan 设计文档、测试基线 82 个全过、后端化(Node+Express+SQLite)设计已就绪。

### 为什么要做 AI 深度融合(合规性)

对照实训评分,今年 AI 相关维度占比很重:能力培养里 **AI 代码审查 30%(新增核心)**;成果评价里 **AI 协作质量 20%**。若平台停留在「采集→存储→展示」的 CRUD 形态,会被判为低价值项目,卡在达标兜底级(上限 85 分),甚至过不了选题终审。

因此本选题在实训期间把 AI 嵌入平台**最痛也最核心**的环节:队员下乡采集回来的是零散照片、录音、口述笔记,「把原始素材整理成规范村庄词条」正是 AI 该干、且天然贯穿主业务的活。

## 2. AI 开发方向清单

按「贴合核心业务 → 增值亮点」排序。实训期间主打**方向一 + 方向二**,方向三及以后为扩展。

### 方向一:AI 驱动的「素材采集 → 结构化词条」管线 ★核心必做

队员上传照片 / 录音 / 口述笔记,AI 自动完成:

1. 语音 / 录音转写成文字。
2. 从口述文本结构化提取:人物故事、村庄简介、年代、资源类型。
3. 自动归类打标签,对接乡村百科已有的六大类标签体系。
4. 生成规范词条初稿 → 队员审校修改 → 入库。

对应实训方向:Web / AI Agent。这是平台命脉,证明 AI 贯穿「采集→入库」核心流程。

### 方向二:乡村知识库 RAG 智能问答 ★技术亮点,冲竞赛级

基于整个资源库做检索增强问答。用户问「湖北有哪些有非遗的村」「哪个村适合亲子研学」,AI 从真实村庄数据检索,给出**带出处**的回答(引用具体村庄词条)。

对应实训方向:AI Agent(明确点名 RAG 检索逻辑)。

### 方向三:AI 智能推荐与「双向匹配」(扩展)

- 村庄智能推荐(按兴趣 / 研学 / 旅游需求)。
- 青年志愿者 ↔ 村庄需求双向匹配(首页 CTA「双向奔赴」已埋概念)。

### 方向四:AI 内容生成辅助(扩展)

- 村庄宣传文案 / 短视频脚本生成。
- 乡村头条:定时抓取三农 / 乡村振兴 RSS 并摘要入库(后端化设计已有此待办)。

### 方向五:AI Agent 自动编排(可选升级)

把采集、结构化、审校、发布做成 Agent 工作流,AI 自动编排多步骤,人在关键节点确认。禁飞区可定为「Agent 核心循环」。

## 3. 技术选型

- **大模型 API**:DeepSeek / 阿里云百炼(实训准备清单指定,免费额度够用)。语音转写用百炼或开源 Whisper。
- **后端**:延续后端化设计的 Node + Express + SQLite + 本地文件存储,单进程国内轻量云服务器。
- **RAG 向量检索**:轻量方案 sqlite-vec 或本地 embedding + 内存向量检索,规模(数十到数百村)绰绰有余,免额外运维。
- **前端**:现有 Vue3 + Vite,新增录入审校界面与问答界面。

选型理由与后端化文档一致:同生态易上手、零配置、成本低、可平滑升级(SQLite→Postgres、本地→OSS、内存向量→专用向量库)。

## 4. 整体架构

```
DigitalVillageInitiative/
├── src/                      # 现有前端 (Vue3)
│   ├── views/AdminIngestView.vue   # 新增:素材上传 + AI 结构化审校界面(方向一)
│   ├── views/AskView.vue           # 新增:RAG 知识问答界面(方向二)
│   └── lib/api.js                  # 统一数据层
├── server/                   # 后端 (Node + Express)
│   ├── ai/
│   │   ├── pipeline.js       # ★方向一:采集→结构化管线编排(禁飞区候选)
│   │   ├── extract.js        # 调用 LLM 做结构化提取 + 打标签
│   │   ├── transcribe.js     # 语音转写
│   │   └── rag.js            # ★方向二:检索+重排+带出处生成(禁飞区候选)
│   ├── routes/
│   │   ├── ingest.js         # 素材上传、AI 结构化、审校保存
│   │   └── ask.js            # RAG 问答
│   └── ...                   # auth / villages / upload 同后端化设计
└── .agent/                   # ★实训要求:项目记忆
    ├── conventions.md
    ├── architecture.md
    └── decisions.md
```

数据流:队员上传素材 → `POST /api/ingest` 触发 `pipeline.js`(转写→提取→打标签→生成词条初稿) → 返回结构化草稿 → 队员审校 → `PUT /api/villages/:id` 入库 → 词条同步进 RAG 索引 → 游客可通过 `AskView` 问答检索。

## 5. 数据模型(在后端化设计基础上增补)

复用 users / villages 两表,新增两表支撑 AI 管线:

### ingest_jobs 表(采集任务)

| 字段 | 类型 | 说明 |
|------|------|------|
| id | INTEGER PK | 主键 |
| village_id | TEXT | 关联村庄(可空,新村待定) |
| raw_media | TEXT | 原始素材路径,JSON 数组 |
| transcript | TEXT | 转写文本 |
| ai_draft | TEXT | AI 生成的结构化词条草稿(JSON) |
| status | TEXT | `pending`/`ai_done`/`reviewed`/`published` |
| created_by | INTEGER | 关联 users.id |
| created_at | TEXT | ISO 时间 |

### village_chunks 表(RAG 索引)

| 字段 | 类型 | 说明 |
|------|------|------|
| id | INTEGER PK | 主键 |
| village_id | TEXT | 来源村庄,用于出处引用 |
| chunk_text | TEXT | 切片文本 |
| embedding | BLOB | 向量(sqlite-vec) |
| source_field | TEXT | 来源栏目(简介/人物/资源等) |

## 6. API 接口(增补 AI 部分)

- `POST /api/ingest/upload` — 上传原始素材(multipart)。需登录。
- `POST /api/ingest/:id/process` — 触发 AI 管线(转写+结构化)。需登录。返回 `ai_draft`。
- `PUT /api/ingest/:id/review` — 保存队员审校结果 → 写入 villages + 更新 RAG 索引。需登录。
- `POST /api/ask` — RAG 问答。body `{ question }`。返回 `{ answer, citations: [{village_id, name}] }`。**公开**。

其余 auth / villages / upload 接口沿用后端化设计。统一错误格式 `{ error }` + HTTP 状态码。

## 7. 对齐实训硬性要求

### AI 禁飞区(3 个核心功能,必须手写、逐行讲解)

1. **RAG 检索 + 重排逻辑**(`server/ai/rag.js`):向量检索、结果重排、出处拼接。
2. **采集管线核心编排**(`server/ai/pipeline.js`):转写→提取→打标签→草稿的多步骤编排。
3. **认证授权模块**(`server/middleware/auth.js`):JWT 签发与校验。

禁飞区代码不用 AI 生成(可参考思路),验收现场逐行讲解,讲不清该模块清零。

## 10. 安全说明

- LLM API Key、JWT 密钥通过环境变量注入,不入库不进仓库。
- 上传接口限文件类型与大小、需登录。
- 问答接口对用户输入做长度与内容约束,防提示词注入滥用。
- 密码 bcrypt 哈希,写接口全经 JWT 鉴权。
