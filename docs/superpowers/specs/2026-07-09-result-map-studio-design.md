## 一、背景与定位

数乡计划是承载乡村社会实践的平台，含六大模块。本次实训要交付、答辩要讲的项目是其中「实践成果」模块下的一个工作台——**成果可视化助手**。

实践队每年采集大量调研数据、访谈、照片、报告，却缺工具把这些原始材料快速整理成可展示、可传播的成果，最终往往只剩一份排版粗糙的 Word/PPT，数据不会说话、成果难以沉淀复用。

成果可视化助手要解决的就是这一段「实践后·出成果」：实践队把材料喂给 AI，AI 在 3D 地图上生成一份「空间叙事」式的可视化成果，队员可直接微调，保存并发布为一件成果作品。

**本期范围（明确收敛）**：只做「实践后·成果可视化工作台」。三阶段 AI 智能体里的「实践前·帮策划」「实践中·督进度」两段已基本实现，本文档不涉及，也不在答辩主线里展开。多形态画布（AI 按意图生成地图/图表/海报）为未来方向，见第八节路线图。

## 二、整体形态与工作流

**页面形态**：左右分栏工作台（`ResultMapStudio.vue`）。

- **左栏 · AI 工作台**：材料上传区 + 对话区。用户上传调研表/访谈/照片/报告，或用自然语言描述实践经过；AI 分析后把结果渲染到右侧地图，并可继续对话调整。输入为「上传打底 + 对话微调」双通道。
- **右栏 · 3D 地图画布**：以**卫星影像**为底图的可视化主体，支持 3D 倾斜/旋转。AI 产出的地图图元长在这里——实践点位标注、走访路线连线、帮扶成效热力、点击点位弹出成果卡片（故事/数据/照片）。卫星底图让用户在拖点校正时能看到真实的屋顶、道路、田块，一眼确认「这个点是不是落在实际去的那个地方」。

**核心工作流（一条闭环）**：

```
上传材料 / 对话描述
      ↓
AI 提取「地点 + 事件 + 数据」→ 地理编码 → 生成地图图元 Spec
      ↓
右侧 3D 地图渲染出点位 / 路线 / 热力 / 弹窗
      ↓
用户 DIY 微调（拖动点位改坐标、实时编辑卡片内容、增删图元）
      ↓
保存 → 发布为「实践成果作品」到成果模块
```

**贯穿始终的一份数据**：一份「成果作品」= 结构化数据（地点、指标、帮扶前后、人物故事、照片）+ 一棵地图可视化 Spec（哪些点、怎么连线、热力字段、每个点的卡片内容）。AI 生成初稿，用户每次微调都直接改这份 Spec/数据，保存的就是它。

**关键取舍**：AI 只产出「受约束的 Spec」，不直接产出渲染代码。这样 AI 输出可被校验、可被用户安全地改，也对应实训「AI 禁飞区 = 手写渲染/校验逻辑」。沿用已有 vizgrammar 语法核思路，只是把 marks 从图表原子换成地图图元。

## 三、地图可视化语法（能画什么）

AI 只能产出下面白名单里的图元，`validateSpec` 校验，手写 renderer 渲染到 3D 地图。本期定义四种地图图元：

| 图元 mark | 是什么 | 需要的数据 | 用户可微调 |
| --- | --- | --- | --- |
| `point`（点位） | 一个实践地点的标注（村庄/活动点） | 坐标、标题、类型 | 拖动改坐标、改标题、改图标/颜色 |
| `route`（路线） | 两个及以上点位间的走访/连线 | 有序点位列表 | 增删途经点、改顺序 |
| `heat`（热力） | 按某指标在地图上铺色（帮扶成效强弱） | 点位 + 数值字段 | 换指标字段、改色阶 |
| `popup`（弹窗卡片） | 点击点位弹出的成果卡：故事/数据/照片 | 绑定某 point + 卡片内容块 | 实时编辑文字、增删数据行/照片 |

**成果作品数据结构（草案）**：

```
{
  title: "小朱湾村电商帮扶实践",
  data: {                          // 结构化成果数据（AI 提取 + 用户订正）
    places: [{ id, name, coord:[lng,lat], type }],
    metrics: [{ placeId, label, value, unit }],
    before_after: [{ placeId, metric, before, after, unit }],
    stories: [{ placeId, title, text, photos:[...] }]
  },
  spec: {                          // 地图可视化 Spec（AI 生成 + 用户 DIY）
    view: { center, zoom, pitch },
    marks: [
      { kind:'point', ref:'place-1', style:{...} },
      { kind:'route', refs:['place-1','place-2'] },
      { kind:'heat',  field:'metrics.value' },
      { kind:'popup', ref:'place-1', blocks:[...] }
    ]
  }
}
```

**关键约定**：

- **地理编码**：AI 提取到地名后需转成坐标。优先匹配平台已有村庄档案的 coord；匹配不到则调地图 API 地理编码；再不行落一个默认点让用户手动拖到正确位置。
- **用户微调直接改这份 JSON**：拖点改 `coord`，编辑卡片改 `stories`/`blocks`，都是就地改数据/Spec，保存即持久化。
- **AI 禁飞区**：地理编码编排、Spec 校验、地图 renderer 三处手写、可逐行讲解。

## 四、AI 层与模块架构

**AI 三步管线**（左栏工作台背后的处理链）：

```
用户输入（上传文件 / 对话）
   ↓
① extract  —— AI 从材料/对话里提取结构化数据
              产出 { places, metrics, before_after, stories }
   ↓
② geocode  —— 地名 → 坐标（手写编排，禁飞区）
              先匹配平台已有村庄 coord → 地图 API 兜底 → 落默认点待用户拖动
   ↓
③ compile  —— AI 把结构化数据编排成一棵地图 Spec（point/route/heat/popup）
              产出的 Spec 必须过 validateSpec 校验
   ↓
validate + render —— 手写校验 + 手写地图 renderer（禁飞区）
```

**模块划分**（延续现有 `src/modules/practice/` 组织，与已有 vizgrammar 平行）：

前端 `src/modules/practice/resultmap/`（纯 JS，全 TDD）：

- `schema.js` —— 地图图元白名单（point/route/heat/popup）+ 每种图元必需字段。约束边界，AI 与 renderer 的唯一契约。
- `validate.js` —— `validateSpec(spec)` 校验 Spec 落在 schema 内。**禁飞区**。
- `geocode.js` —— 地名→坐标编排（匹配村庄库 / API / 默认点）。**禁飞区**。
- `renderer.js` —— 遍历 Spec 树，产出 3D 地图图层配置。**禁飞区**。
- `ResultMapStudio.vue` —— 工作台页面：左 AI 栏 + 右 3D 地图，接上传/对话/微调/保存。

后端 `server/services/`：

- `resultExtractService.js` —— 封装 extract（材料/对话 → 结构化数据），复用现有 deepseek.js + 限流。
- `resultCompileService.js` —— 封装 compile（结构化数据 → Spec）。
- 作品的存取走 SQLite（新增 result_works 表）+ 文件上传（延续既定后端化方案）。

**地图技术选型**：用 **高德地图 JS API + 卫星图层**。理由——本工作台的核心价值之一是「拖点校正」，用户要能确认 AI 落的点是不是自己实际去的地方；卫星影像能看到真实地物（屋顶、道路、田块），矢量底图做不到。高德 JS API 原生支持卫星图层、标记点、拖动标记、3D 倾斜/旋转（pitch/rotate），且项目已有高德账号体系（env 里已有 `AMAP_API_KEY`）。

- **代价**：引入高德 JS SDK（新依赖）；renderer 从「组装 ECharts option」变为「操作高德地图实例（增删图层/覆盖物）」。首页原有的 ECharts 地图不受影响，本工作台独立用高德。
- **Key 说明**：现有 `AMAP_API_KEY` 是**高德 Web 服务 Key**（服务端海拔查询用）。浏览器端显示卫星图需另申请一个**高德 JS API Web 端 Key**（含域名白名单），见开放问题。

**依赖方向**：`schema ← validate`、`schema ← geocode`、`schema ← renderer`、`ResultMapStudio → validate/renderer/geocode + 后端 service`。无循环依赖。

## 五、异常处理

- **AI 无 key / 超时 / 失败**：沿用 deepseek.js 的 NoKeyError/DeepseekError 兜底——extract/compile 失败给出明确提示，工作台保留用户已有材料不丢失，允许重试。
- **AI 产出非法 Spec**：`validateSpec` 拦截，不渲染坏图元，提示「AI 生成的这部分有问题，已跳过」，用户可手动补。既是安全闸也是防注入。
- **地理编码失败**：匹配不到坐标的地点落一个默认点 + 高亮提示「位置待确认」，引导用户拖到正确位置。
- **空/脏数据**：材料提取不出结构化内容时，明确告知「未能识别出地点/数据」，而非静默生成空地图。

## 六、测试策略

延续项目 Vitest + TDD 基线，勿回归。

- `schema/validate/geocode/renderer` 四个纯函数模块全 TDD 覆盖（合法/非法 Spec、坐标匹配兜底、各图元渲染）。
- 端到端集成测试：一份样例材料 → extract → compile → validate → render 一条链跑通。
- 后端 service 层测试用依赖注入的假 fetch，不打真 API。

## 七、验收标准

1. 上传一份含地点与数据的材料（或对话描述）→ AI 在 3D 地图上生成点/路线/热力/弹窗。
2. 用户能拖动点位改坐标、实时编辑弹窗卡片内容、增删图元。
3. 保存后作品持久化，可在实践成果模块查看/再编辑。
4. 非法 AI 输出被 validate 拦下、地理编码失败有兜底、AI 失败不丢材料。
5. `npm test` 全绿，四个纯函数模块 + 集成测试覆盖，无回归。

- **更强多模态**：照片内容理解、更深的材料解析。
