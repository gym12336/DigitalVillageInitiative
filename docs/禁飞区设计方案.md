- 每个项目设定 **3 个核心功能** 为"AI禁飞区"，这些功能必须**手写**

## 三、禁飞区①：认证授权全链路（✅ Sprint1完成）

### 3.1 定位

对应官方示例方向「**认证授权模块**」。覆盖用户注册 → 登录 → JWT签发 → 请求鉴权的完整闭环，不依赖任何第三方认证框架（仅用 `jsonwebtoken` 和 `bcryptjs` 做底层密码学运算）。

### 3.2 文件与职责

```
server/
├── lib/token.js           ← JWT 签发/校验（21行）
├── middleware/auth.js      ← Bearer提取 + 验签 + 挂载req.user（29行）
└── services/userService.js ← 注册/登录 + bcrypt密码哈希（64行）
```

### 3.3 数据流

```
注册: validateRegister → bcrypt.hash(password,10) → INSERT users → signToken → 返回{token,user}
登录: validateLogin → SELECT user → bcrypt.compare → signToken → 返回{token,user}
请求: Authorization头部 → 正则提取Bearer token → jwt.verify → SELECT确认用户存在 → req.user
```

### 3.4 逐行讲解要点

#### token.js（21行）

| 行 | 要点 | 讲解 |
|----|------|------|
| 5 | `EXPIRES_IN = '7d'` | 7天过期，平衡安全与体验 |
| 8-11 | `signToken(userId, secret)` | 签发token，`secret`显式传入——不从全局读，便于单测 |
| 10 | `jwt.sign({ userId }, secret, { expiresIn })` | 载荷只存 `userId`，不塞权限/角色——保持token轻量 |
| 11 | `if (!secret) throw` | 防御性编程：secret缺失立即报错，不静默失败 |
| 17-21 | `verifyToken(token, secret)` | 校验token，`jwt.verify`失败自动抛错（签名错/过期/格式错） |
| 20 | `return { userId: payload.userId }` | 只返回userId，调用方（中间件）决定怎么用 |

#### middleware/auth.js（29行）

| 行 | 要点 | 讲解 |
|----|------|------|
| 10 | `makeAuth(secret, db)` | **工厂函数**：依赖注入secret和db，单测时可换内存数据库 |
| 12-14 | `header.match(/^Bearer\s+(.+)$/i)` | 正则提取Bearer token，`i`标志不区分大小写 |
| 14 | `if (!m) return next(httpError(401))` | 无token → 401，不暴露内部细节 |
| 16-21 | `verifyToken(m[1], secret)` | 调token.js校验，失败catch → "登录已失效" |
| 23-24 | `db.prepare(...).get(userId)` | **关键安全设计**：验签通过后再查库，防token有效但用户已被删除 |
| 26 | `req.user = { id, username }` | 只挂身份，不挂teamId——多队场景下归属交给service层 |
| 27 | `next()` | 通过则放行，不做多余的事 |

#### userService.js（64行）

| 行 | 要点 | 讲解 |
|----|------|------|
| 7 | `BCRYPT_COST = 10` | 10轮哈希，安全与性能的平衡点 |
| 10-19 | `teamsForUser(db, userId)` | 联表查memberships，返回用户加入的所有队伍及角色 |
| 22-29 | `toPublicUser(db, row)` | **安全保障**：绝不返回password字段；teams为空数组而非null |
| 41-52 | `register(db, {username,password,displayName})` | 先查用户名是否已存在（冲突→409），然后bcrypt哈希，写库，返回公开user |
| 58-63 | `login(db, {username, password})` | 查用户 → bcrypt比对；**用户名不存在和密码错误统一报401**（安全原则：不泄露哪一项错） |
| 60-62 | 统一401 | "用户名或密码错误"——攻击者无法判断是否存在该用户 |

## 四、禁飞区②：AI输出防火墙（planSchema.js ✅ Sprint1完成）

### 4.1 定位

**这是系统信任AI的边界线。** LLM生成的JSON方案，必须经过 `validatePlanShape()` 的结构校验，不合格直接丢弃、走本地规则引擎兜底。它和 planTemplate.js 共同构成「AI + 规则」双引擎。

### 4.2 核心函数：`validatePlanShape(input)`（107行）

```
LLM返回JSON → validatePlanShape → { ok: true, plan } 或 { ok: false, reason }
```

### 4.3 三层校验链

| 层级 | 校验内容 | 不通过后果 |
|------|---------|-----------|
| **第一层** | 顶层必须是object，`phases`必须是数组 | 返回 `{ok:false, reason:'not-object'/'phases-missing'}` |
| **第二层** | 三段（plan/track/result）缺一不可 | 返回 `{ok:false, reason:'stage-missing:plan'}` 等 |
| **第三层** | 每条task必须有非空text，无效task剔除，未知字段丢弃 | 静默过滤，保证输出干净 |

### 4.4 逐行讲解要点

| 行 | 要点 | 讲解 |
|----|------|------|
| 4-5 | `STAGES = ['plan','track','result']` | 恒三段，对应实践前/中/后，不接受LLM自由发挥 |
| 61-64 | 第一层校验 | `typeof input !== 'object'` / `!Array.isArray(input.phases)` → 直接不合格 |
| 70-87 | 第二层校验 | 遍历phases，`stageSeen`记录出现过的阶段，缺任何一个stage → 不合格 |
| 73-74 | `if (!STAGES.includes(p.stage)) continue` | 未知阶段直接跳过——LLM可能编造"review"之类的阶段 |
| 78-81 | 第三层校验 | 每个task调`normTask`，text为空就剔除——不把LLM的空槽位写进数据库 |
| 89-91 | 三段完整性检查 | `for (const s of STAGES) { if (!stageSeen.has(s)) return ... }` |
| 95-105 | 规范化输出 | 所有字段过`normStr`/`normStringArray`/`normMetrics`，确保输出类型稳定 |

## 五、禁飞区②-ext：离线规则引擎（planTemplate.js 📋 设计先行）

> **本期状态：设计已完成，代码Sprint2实现。** 当前LLM不可用时由 planService.js 中的 `generateTemplatePlan` 调用产生方案，planTemplate.js 作为独立禁飞区模块将在Sprint2重构抽取。

### 5.1 定位

AI输出防火墙的**另一半**——当校验失败、LLM超时、无API Key时，由本地规则引擎产出合法方案，保证系统永远可用。是项目"AI辅助但不依赖AI"思想的核心体现。

### 5.2 设计方案

```
planTemplate.js 结构:

1. 选题库 TOPICS[]（4类 × 约30行/类 ≈ 130行）
   - 文化类：关键词[竹编, 非遗, 手艺...] → 三阶段任务模板
   - 产业类：关键词[电商, 品牌, 直播...] → 三阶段任务模板
   - 教育类：关键词[支教, 课程, 留守...] → 三阶段任务模板
   - 生态类：关键词[旅游, 环保, 规划...] → 三阶段任务模板
   - DEFAULT_TOPIC：兜底选题

2. 核心算法
   pickTopic(idea, refs, topicHint)
   - 在idea + refs文本中扫描关键词
   - 第一个命中的作为主选题
   - 无命中 → DEFAULT_TOPIC

   pickTargetVillage(idea, refs)
   - 正则提取"XX村"，过滤"乡村/农村/山村"泛称
   - 优先从retrieval命中的村庄卡片取

   generateTemplatePlan(input)
   - 选题匹配 → 拼装goal/metrics/methods/risks/phases
   - 输出结构与AI版完全同构
   - source: 'template' 标记来源

3. 设计原则
   - 纯函数，零外部依赖
   - 输出结构通过 validatePlanShape 校验（与AI版同一套标准）
   - source字段区分 'ai' / 'template'，可追溯方案来源
```

### 5.3 安全价值

- **网络全断也能出方案**：不依赖任何外部服务
- **不产生幻觉**：选题库是领域知识沉淀，每条任务真实可执行
- **与AI版同构**：对调用方（前端/数据库）透明，不需要区分来源

## 六、禁飞区③：LLM安全通道 + 可视化双核（📋 设计先行 / Sprint2实现）

> **本期状态：deepseek.js已完成174行基础实现，compile.js和renderer.js设计已完成，代码Sprint2实现。**

### 6.1 禁飞区③-1：LLM安全通道（deepseek.js）

#### 定位

项目对接AI的**唯一入口**。API Key隔离在服务端，前端不可见。所有LLM调用必须经过此模块。

#### 已完成的核心设计

| 安全措施 | 实现 |
|---------|------|
| API Key隔离 | `apiKey = process.env.DEEPSEEK_API_KEY`，仅服务端读取 |
| 超时控制 | `AbortController` + 20秒超时 |
| 单次重试 | 网络抖动容错，不重试NoKeyError |
| 强制JSON | `response_format: { type: 'json_object' }` |
| 错误分类 | `NoKeyError`（配置问题）vs `DeepseekError`（服务问题） |
| 速率限制 | `TokenBucket`令牌桶（5次/秒），`rateLimiter.js`配合 |

### 6.2 禁飞区③-ext：可视化双核（compile.js + renderer.js）

#### 定位

"约束下的自由组合"的技术脊梁。基于 **Grammar of Graphics**（图形语法，Vega-Lite/ggplot思想），两个手写核心构成闭环：

```
compile.js（意图→Spec编译器）  →  Spec JSON  →  renderer.js（Spec→ECharts渲染引擎）
     NL意图+数据+Schema约束           │              纯函数，可单测
     AI只在Schema内组合                │              每个mark一个渲染分支
     不吐可执行代码                   │
                              validate.js
                            （结构校验，防注入）
```

#### 设计方案

**compile.js** —— 意图→Spec编译器
```
输入: { intent: "突出帮扶后收入翻倍", data: { before_after: [...] } }
       ↓
1. 意图分类: 关键词匹配 → [变化/概览/过程/构成/人物/空间]
2. Schema查候选mark: 变化类 → [dumbbell, bar-compare, slope, delta-badge]
3. 数据匹配: 检查data中是否有mark需要的字段
4. 组装Spec: { layout, blocks: [{ slot, kind, mark, encoding, annotations }] }
输出: 符合schema的可视化Spec JSON
```

**renderer.js** —— Spec→ECharts渲染引擎
```
输入: Spec JSON
       ↓
1. 遍历 blocks[]
2. 按mark类型分发: dumbbell→renderDumbbell, timeline→renderTimeline, ...
3. 每个mark翻译成ECharts option
4. 组装布局
输出: 完整ECharts配置对象
```

**一期mark集合（4种）**：
| mark | 叙事意图 | 数据要求 |
|------|---------|---------|
| `dumbbell` | 讲变化（帮扶前后对比） | before_after |
| `stat-row` | 讲概览（KPI卡片） | metrics |
| `timeline` | 讲过程（时间轴） | timeline |
| `people-card` | 讲人与故事 | people |

**核心契约：可视化Spec**

Spec不是模板ID，是一棵用原子零件拼出来的组合树。compile输出和renderer输入由同一Schema约束，二者独立可测、独立可讲：

```json
{
  "layout": { "type": "grid", "slots": ["hero", "body-left", "body-right"] },
  "blocks": [
    {
      "slot": "hero",
      "kind": "chart",
      "mark": "dumbbell",
      "encoding": {
        "y": { "field": "指标名", "from": "before_after" },
        "x1": { "field": "前值" }, "x2": { "field": "后值" }
      },
      "annotations": [{ "at": "收入", "text": "翻倍", "emphasis": true }],
      "palette": "earth-green"
    }
  ]
}
```

#### 可测试性

- `compile.js`：给定意图+数据，断言输出Spec的mark/encoding关键字段
- `renderer.js`：纯函数，给定Spec→断言ECharts option结构
- `validate.js`：合法/非法Spec边界用例

## 七、禁飞区与AI的协作边界

```
                    ┌──────────────────────────────┐
                    │        AI 可以参与的区域        │
                    │  · 方案生成（经schema校验）     │
                    │  · 结构化信息提取             │
                    │  · 可视化Spec编排（Schema内）  │
                    │  · 叙事文案生成               │
                    └──────────────┬───────────────┘
                                   │ AI输出
                    ┌──────────────▼───────────────┐
                    │        禁飞区（手写）          │
                    │  · 认证授权（token+auth+user） │
                    │  · 输出校验（planSchema）      │
                    │  · 离线兜底（planTemplate）    │
                    │  · LLM通道（deepseek）         │
                    │  · Spec编译+渲染（compile+render）│
                    └──────────────────────────────┘
                                   │
                    "AI只吐结构化数据，不吐可执行代码"
                    "约束下的自由组合"
```
