# AI 抗性挑战 · 架构重构

> 挑战要求：选项目中最复杂的函数，圈复杂度降低 50% 且行为不变，附前后对比。
> 完成人：gym（高一鸣）。

## 一、如何选定"最复杂的函数"

不靠感觉，用圈复杂度（Cyclomatic Complexity，CC）量化。CC = 1 + 分支判定点数量（`if / else if / for / while / case / && / || / ?: / ?.` 等）。CC 越高，独立执行路径越多，越难测、越易藏 bug。

对候选源文件逐函数统计，结果排名前列：

| 函数 | 文件 | CC | 行数 |
|---|---|---|---|
| **`calcLayoutSlots`** | `layoutBoxEngine.js` | **37** | 150 |
| `retrieve` | `retrieval.js` | 29 | 100 |
| `normalizeFacts` | `villageService.js` | 24 | 18 |
| `renderChartSvg` | `chartRenderer.js` | 17 | 45 |

`calcLayoutSlots` 是全项目圈复杂度最高的函数（CC=37），选它作为重构目标。

它的职责是：给成果搭建台的"多组件框"计算每个 slot 的矩形位置和分隔条位置。原实现是一个 **12 分支的巨型 `switch`**，每个 case 内又有循环和三元表达式，路径极多。

## 二、重构方法：巨型 switch → 数据驱动分派

### 重构前（CC=37）

一个函数里塞了 12 种布局的全部计算，`switch(layout)` 每个 `case` 手写坐标：

```js
export function calcLayoutSlots(containerW, containerH, layout, splitRatios, slotCount) {
  const innerW = containerW - PAD * 2
  const innerH = containerH - PAD * 2
  const slots = []
  const dividers = []
  switch (layout) {
    case 'horizontal': { /* 循环 + 三元 + 分隔条 */ break }
    case 'vertical':   { /* 循环 + 三元 + 分隔条 */ break }
    case 'grid-2x2':   { /* 4 slot + 2 分隔条 */ break }
    case 'main-right': { /* ... */ break }
    // …共 12 个 case + default 回退…
  }
  return { slots, dividers }
}
```

问题：所有布局的逻辑耦合在一个函数里，改任何一种布局都要在 150 行里翻找；测试也难以定位失败在哪种布局。

### 重构后（CC=2）

主函数只做三件事：建几何上下文 `ctx` → 按 `layout` 查分派表 → 未知键回退 `horizontal`。每种布局拆成独立的低复杂度生成器：

```js
const LAYOUT_BUILDERS = {
  'horizontal': (ctx) => buildEqualSplit(ctx, 'h'),
  'vertical':   (ctx) => buildEqualSplit(ctx, 'v'),
  'grid-2x2':   buildGrid2x2,
  'main-right': (ctx) => buildSplit2(ctx, 'h', 67),
  'main-left':  (ctx) => buildSplit2(ctx, 'h', 33),
  'main-bottom':(ctx) => buildSplit2(ctx, 'v', 67),
  'main-top':   (ctx) => buildSplit2(ctx, 'v', 33),
  '1+2-right':  build1plus2Right,
  '2+1-right':  build2plus1Right,
  '1+2-bottom': build1plus2Bottom,
  '2+1-top':    build2plus1Top,
}

export function calcLayoutSlots(containerW, containerH, layout, splitRatios, slotCount) {
  const ctx = {
    PAD, DIV,
    innerW: containerW - PAD * 2,
    innerH: containerH - PAD * 2,
    containerW, containerH, splitRatios, slotCount,
  }
  const build = LAYOUT_BUILDERS[layout] || ((c) => buildEqualSplit(c, 'h'))
  return build(ctx)
}
```

关键手法：
1. **消除重复**：`horizontal`/`vertical`/`default` 原本是三段几乎一样的等分循环，合并成一个 `buildEqualSplit(ctx, dir)`，方向用参数区分。
2. **消除重复**：`main-right/left/bottom/top` 四种"二分"布局合并成 `buildSplit2(ctx, dir, firstDefault)`。
3. **switch → 查表**：`LAYOUT_BUILDERS` 对象把布局键映射到生成器，主函数不再有分支。未知布局回退用 `|| ` 兜底，与原 `default` 行为一致。

## 三、前后对比（圈复杂度）

| 指标 | 重构前 | 重构后 | 变化 |
|---|---|---|---|
| **`calcLayoutSlots` CC** | **37** | **2** | **↓ 95%** |
| 主函数行数 | 150 | 12 | ↓ 92% |

拆出的 7 个生成器函数，各自 CC 都很低、职责单一：

| 生成器 | CC | 覆盖的布局 |
|---|---|---|
| `buildEqualSplit` | 7 | horizontal / vertical / 未知回退 |
| `buildGrid2x2` | 5 | grid-2x2 |
| `buildSplit2` | 4 | main-right/left/bottom/top |
| `build1plus2Right` | 3 | 1+2-right |
| `build2plus1Right` | 3 | 2+1-right |
| `build1plus2Bottom` | 3 | 1+2-bottom |
| `build2plus1Top` | 3 | 2+1-top |

CC 从 37 降到 2，**降幅 95%，远超挑战要求的 50%**。且没有把复杂度"藏"到别处——最复杂的生成器 CC 也只有 7。

## 四、如何证明"行为不变"

重构最大的风险是悄悄改变了输出。为此采用**黄金测试（characterization test）**：

1. **重构前**：用旧实现对 800×600 容器跑遍 12 种布局 + fallback + 空 ratios，把每种的 `{ slots, dividers }` 输出固化成基准（golden）。
2. 写成测试 `src/__tests__/builder-layoutBoxEngine.test.js`（21 个用例），**在重构前先跑绿**，证明它准确捕获了当前行为。
3. **重构后**：同样的测试必须逐字节一致（`toEqual`）。

结果：

- 黄金测试 **21/21 全绿**（重构前后都绿，输出完全一致）。
- 全量回归 **70 个测试文件、528 个测试全部通过**，零回归。

行为不变不是靠"我觉得没改"，而是靠测试逐字节比对证明的。
