## 二、Review 结论（9 条建议）

### 建议 1 · `panMath.js` 换算公式清晰，建议补充极区边界测试【已确认，低优先】

- **观察**：`pixelsToLngLat` 用 Web Mercator 公式，`mpp = 156543.03392 * cos(latRad) / 2^zoom`，经度换算除以 `111320 * cos(latRad)`，纬度除以 `110540`。公式正确，且已有 6 个单测覆盖赤道、不同纬度、不同 zoom、方向。
- **潜在问题**：`cos(latRad)` 在纬度接近 ±90°（极区）时趋近 0，`dLng` 会被放大到异常值。中国境内不会触发，但作为纯函数缺一个极区兜底。
- **建议**：低优先。可加一条 `centerLat=85` 的测试固化当前行为，或在文档注明"仅适用于非极区"。属加固，非缺陷。

### 建议 2 · `buildPreview.js` 跨标签页传 state 用全局变量，需注意生命周期【已确认，中优先】

- **观察**：`buildAndOpen` 把 state 存到 `window.__previewState[id]`，再 `window.open('#/builder/preview?id='+id)` 开新标签；`PreviewStage.vue` 用 `window.opener.__previewState[id]` 读取后立即 `delete`。
- **潜在问题**：这依赖新标签页能访问 `window.opener`。若用户手动复制预览 URL 到全新标签打开（无 opener），或原标签页已关闭，state 就丢了。
- **现状**：`PreviewStage` **已有兜底**——读不到就显示"预览已失效，请回到编辑器重新预览"，不会崩。处理得当。
- **建议**：中优先。当前兜底合格。若要更健壮，可考虑 `sessionStorage` 传递（跨标签、可刷新），但会引入清理成本，当前方案对"编辑器→预览"主路径够用。

### 建议 3 · `tianditu.js` 的 API 契约本次已由 gym 修复,建议 alice 同步认知【已修复】

- **背景**：alice 的 `ce028f0` 已把搜索改成 v2 的 `postStr` JSON 格式，方向正确。但参数名仍是 `specify`+6 位码、成功码只认 `status==='0'`、坐标读 `poi.lon/lat`。
- **本次修复**（gym，提交 `65f5e1b`）：改为 `specifyAdminCode`+9 位国标码（`156`前缀）、兼容 `status.infocode===1000` 对象格式、新增 `parseLngLat` 解析 `lonlat` 字段。详见 [AI反思日志.md](AI反思日志.md) 第 1 条。
- **建议**：alice 同步这次契约修正的认知，后续接天地图其他接口（如逆地理编码）注意同样的 v2 差异。

### 建议 4 · `v-html` 渲染路径已确认做了转义，安全性合格【已确认无问题】

- **观察**：预览组件对 chart/timeline/datatable 用 `v-html` 渲染。gym 追查了渲染器源码。
- **结论**：`datatableRenderer.js` 和 `timelineRenderer.js` 都有 `esc()` 函数，对用户文本做了 `& < > "` 的 HTML 转义后才拼进 innerHTML。**不存在 XSS 注入风险**，这点做得好。
- **建议**：保持。后续新增任何 v-html 渲染器都必须走同样的 `esc()`。

### 建议 5 · map3d 依赖 Cesium 体积大,建议确认按需加载【已确认，架构合理】

- **观察**：`cesiumScene.js` 被 vitest 覆盖率显式排除；`vite.config.js` 为 cesium 单独分 chunk，`build` 有 `copy-cesium-assets` 的 postinstall。
- **潜在关注**：Cesium 是重依赖，若首页或非 3D 页面同步加载会拖慢首屏。
- **结论**：alice 的 `7fe7107` 提交用 **dynamic import** 动态引入 Cesium（"map-3d preview HTML generation with dynamic Cesium import"），只在预览页按需加载，不影响首页。**架构合理**。
- **建议**：保持。性能门禁测首页加载时，确认 3D 相关 chunk 未进首屏 bundle。

### 建议 6 · `VillageSearchField` 已加 300ms 防抖，避免搜索请求风暴【已确认无问题】

- **观察**：早期版本每次按键都触发天地图搜索，`50e65c4` 提交加了 300ms 防抖（`debounceTimer` + `setTimeout`），并在卸载/选中时 `clearTimeout` 清理。
- **结论**：防抖 + 清理都到位，不会因连续输入打爆天地图配额，也没有卸载后回调的泄漏。做得好。
- **建议**：保持。

### 建议 7 · `layoutBoxEngine.calcLayoutSlots` 抽成纯函数,可测性好【已确认，架构合理】

- **观察**：多组件框的分栏尺寸/slot 定位计算被抽到 `layoutBoxEngine.js` 的纯函数 `calcLayoutSlots(containerW, containerH, layout, splitRatios, slotCount)`，与渲染分离（`7acb487`）。
- **结论**：纯函数化让布局逻辑能脱离 DOM 单测，也让 EditorCanvas 和预览复用同一套计算。这是好的边界设计。
- **建议**：低优先——`calcLayoutSlots` 目前的单测覆盖建议补充"比例和不为 1""slotCount 为 0"两个边界，防止后续改动回归。

### 建议 8 · sankey CSV 解析对坏输入返回空数组,不崩【已确认无问题】

- **观察**：`chartRenderer.js` 的 sankey 解析（`c1ce387`）对行数不足（`lines.length < 2`）、缺列（`labelIdx === -1` 等）都 `return []`，不会因为用户 CSV 格式不对而抛异常。
- **结论**：坏数据降级为空图而非崩溃，符合"损坏输入不崩"的原则。
- **建议**：低优先——可给用户一个"CSV 格式不正确"的可见提示，目前是静默返回空图，用户可能不知道为什么没渲染。属体验优化，非缺陷。

### 建议 9 · `PracticeImagePicker` 有卸载清理与重试,健壮性好【已确认无问题】

- **观察**：`aede240` 起的 PracticeImagePicker 有 `onBeforeUnmount` 清理、失败时的"重试"按钮（`b758f17`），并修过材料数据路径（`7d32c1e`）。
- **结论**：异步取图 + 卸载清理 + 失败重试，覆盖了组件生命周期的边界，不会卸载后 setState 报警告。
- **建议**：保持。
