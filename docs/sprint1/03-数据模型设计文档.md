## 1. 存储策略：标量列 + JSON payload

实践档案采用 **「关键字段提列、主体整存 JSON」** 的混合模式。

**为什么这样设计？**
实践档案的结构会随三阶段（plan/track/result）不断演进，字段多且嵌套深（方案、指标、材料、人物、综述……）。若全部拆成关系表，schema 会非常臃肿且频繁变更。所以：
- **需要查询/排序/展示的少数字段** → 提成标量列（`team_id`、`title`、`stage`、`updated_at`）；
- **完整档案主体** → 整存一列 `payload`（JSON 文本）。

这样列表页靠标量列高效检索，档案细节靠 `payload` 灵活演进，改结构不用改表。

## 2. 核心表：dossiers（实践档案）

```sql
CREATE TABLE IF NOT EXISTS dossiers (
  id          TEXT PRIMARY KEY,                        -- 前端 genId 生成的字符串 id
  team_id     INTEGER NOT NULL REFERENCES teams(id),   -- 归属队（团队共享）
  created_by  INTEGER NOT NULL REFERENCES users(id),   -- 建档人（审计 + 删除授权）
  title       TEXT NOT NULL DEFAULT '',                -- 从 payload 冗余提升，供列表展示
  stage       TEXT NOT NULL DEFAULT 'plan',            -- 'plan'|'track'|'result'
  payload     TEXT NOT NULL,                           -- 完整 dossier JSON
  created_at  TEXT NOT NULL,
  updated_at  TEXT NOT NULL
);

-- 列表页按本队 + updated_at 倒序取，复合索引更顺。
CREATE INDEX IF NOT EXISTS idx_dossiers_team_updated ON dossiers(team_id, updated_at);
```

**字段说明与取舍：**

| 字段 | 说明 | 为什么 |
|------|------|--------|
| `id` | 字符串主键 | 沿用前端 `genId()` 生成，前端离线也能先建档，避免依赖自增 id 往返 |
| `team_id` | 归属队 | 档案是**团队共享**的，同队成员都能看/传材料；不是归属个人 |
| `created_by` | 建档人 | 仅用于审计和删除授权（谁建的谁能删），不影响共享 |
| `title` / `stage` | 从 payload 冗余提升 | 列表页要展示标题、要按阶段筛选，提列避免每行解析 JSON |
| `payload` | 完整档案 JSON | 主体，实践中采集的所有数据都在这里的 `collected` 字段下 |

**复合索引 `(team_id, updated_at)`**：列表页查询固定是"取本队档案、按更新时间倒序"，复合索引让 `WHERE team_id=? ORDER BY updated_at DESC` 走索引不回表排序。

## 3. payload JSON 结构（实践中相关部分）

`payload` 反序列化后是完整 dossier 对象。实践中阶段读写的核心是 `collected` 子对象。以下结构从 [StageTrack.vue](../../src/modules/practice/mine/StageTrack.vue) 的 `save()`/`clone()` 精确还原：

```jsonc
{
  "id": "dmre7qridjjnq",
  "title": "小朱湾村电商帮扶",
  "stage": "track",
  "village": "小朱湾村",
  "startDate": "2026-07-06",       // 督进算时段进度用
  "endDate": "2026-07-20",

  "plan": {                         // 实践前生成的方案（实践中只读对照）
    "topic": "电商帮扶",
    "targetVillage": "小朱湾村",
    "metrics": [                    // 计划采集的指标 → 实践中对照填值
      { "name": "季度销量", "unit": "元" }
    ],
    "phases": [
      { "stage": "track",
        "tasks": [ { "text": "走访3户种植户", "output": "访谈记录", "done": true } ] }
    ]
  },

  "collected": {                    // ⭐ 实践中采集的数据，本模块的写入主体
    "metricValues": [
      { "name": "季度销量", "before": "5000", "after": "12000",
        "unit": "元", "insight": "增长明显", "isHighlight": true }
    ],
    "materials": [
      { "type": "文档", "name": "调研报告.docx", "note": "",
        "url": "/uploads/practice/<id>/xxx.docx",
        "kind": "doc", "ext": "docx",
        "summary": "一句摘要", "theme": "产业帮扶" }
    ],
    "people": [
      { "name": "李大爷", "role": "返乡创业带头人", "quote": "……",
        "story": "2-3句故事", "highlight": "返乡创业带头人" }
    ],
    "summary": "本次实践成果综述……",
    "highlights": ["亮点1", "亮点2"]
  }
}
```

### 3.1 collected 三大数组的字段来源

三个数组的字段和 AI 抽取契约（[extractPrompt.js](../../server/services/prompts/extractPrompt.js)）一一对应，**采纳草稿时字段直接对齐**：

| 数组 | 字段 | 事实/加工 |
|------|------|----------|
| `metricValues` | `name, before, after, unit` | 事实（`before` 现场填，`after` 可由 AI 的 `value` 采纳而来） |
| | `insight, isHighlight` | 加工（AI 解读、是否重点） |
| `materials` | `type, name, note, url, kind, ext` | 存储元数据（`url/kind/ext` 来自 `storeFile`） |
| | `summary, theme` | 加工（AI 摘要、主题归类） |
| `people` | `name, role, quote` | 事实 |
| | `story, highlight` | 加工（AI 生成人物故事、亮点标签） |

**设计要点：AI 抽取产出（人物有 name+quote、指标有 name+value）经审校采纳后，字段结构与 collected 完全对齐**，采纳即 `push` 无需转换（见 `adoptDraft`）。

### 3.2 待审校草稿（draft）—— 不落库的中间态

AI 抽取的结果**不直接进 payload**，先进前端内存的 `draft`（人物/指标/材料要点三组），队员逐条采纳才写入 `collected`。所以：
- `draft` 是**纯前端态**，刷新即弃，不持久化；
- 每条草稿带 `source:'auto'`（标记 AI 产出）、`confidence`（把握度）、`sourceFile`（来自哪个上传文件，ZIP 导入时标注）；
- 采纳时剥离 `source/confidence/sourceFile`，只留业务字段进 `collected`。

这落实了"AI 是辅助、人做决策、只有人工确认过的才入库"的合规原则。

## 4. 材料文件的落盘约定

材料文件**不存数据库**，只存磁盘，`collected.materials[].url` 记路径。

```
server/uploads/practice/
  └─ <dossierId>/            按档案分目录
       ├─ dmred1o4qdiem.png  存储名 = genId() + 原扩展名（重铸，防路径穿越/覆盖）
       └─ dmredplbyknve.docx
```

- **分目录**：按 `dossierId` 隔离，一份档案的材料聚在一起。
- **重铸存储名**：用 `genId()` 生成新名 + 保留原扩展名，绝不用用户原始文件名落盘（防路径穿越、防重名覆盖）。原文件名存在 `materials[].name` 供展示。
- **访问路径**：`/uploads/practice/<dossierId>/<存储名>`，由 `express.static` 只读托管。

**分档大小上限**（`mediaService.SIZE_LIMITS`）：

| kind | 扩展名 | 上限 |
|------|--------|------|
| `av` 音视频 | mp4/mov/mp3/wav… | 200MB |
| `image` 图片 | jpg/png/webp/gif… | 20MB |
| `doc` 文档 | txt/md/docx/pdf | 10MB |
| `table` 表格 | csv/xlsx/xls | 10MB |
| `other` 其他 | — | 20MB |

分档理由：音视频天然大，文本档小；按类型给不同上限，既不误拦大视频，也不让文本档滥用空间。

## 5. AI 抽取的数据契约

`practiceExtractService` 对 LLM 返回做**规范化**，保证进 `draft` 的每条结构合法：

```js
// 人物规范化：补空串、默认 confidence 0.6、标 source:'auto'
normPerson(p) → { name, role, quote, story, highlight, confidence, source:'auto' }
normMetric(m) → { name, value, unit, insight, isHighlight, confidence, source:'auto' }
normHint(h)   → { name, note, summary, theme, confidence, source:'auto' }
```

- 数字字段（`confidence`）用 `Number.isFinite` 兜底，非法给默认 0.6；
- `value/unit` 一律转字符串（`str()`），避免类型不一致；
- LLM 漏字段 → 补空串；返回非数组 → 给 `[]`。**永远不把脏数据带进前端。**

## 7. 补充：乡村百科的 villages 表

乡村百科的村庄数据同样采用 **「标量列 + JSON 列」** 混合模式，与 dossiers 思路一致。

```sql
CREATE TABLE IF NOT EXISTS villages (
  id           TEXT PRIMARY KEY,               -- 如 'chenjiapu'
  name         TEXT NOT NULL,
  province     TEXT NOT NULL DEFAULT '',        -- 提列：筛选/聚合
  city TEXT, district TEXT, town TEXT,
  summary      TEXT, intro TEXT, cover TEXT,    -- 摘要/介绍/封面
  cert_level   TEXT DEFAULT 'township',         -- 认证等级
  coord_lat REAL, coord_lng REAL,               -- 经纬度（地图打点）
  views INTEGER, favorites INTEGER, practices INTEGER,  -- 计数
  -- JSON 文本列（内容可变、不单独建索引）
  honors TEXT DEFAULT '[]',                      -- ["中国传统村落",...]
  tags   TEXT DEFAULT '{}',                      -- {"文化类":[...],...}
  gallery TEXT, guide TEXT, socials TEXT, manager TEXT,
  facts TEXT, timeline TEXT, specialties TEXT, festivals TEXT, sections TEXT,
  created_at TEXT, updated_at TEXT
);
CREATE INDEX idx_villages_province ON villages(province);
CREATE INDEX idx_villages_name ON villages(name);
```

**要点：**
- **标量列 vs JSON 列**：可查询/排序/打点的字段（`province`/`name`/`coord`/计数）提列并建索引；富内容（`tags`/`honors`/`facts`/`sections` 等）存 JSON 文本。
- **读取解析**：`villageService.rowToVillage` 把 JSON 列 `JSON.parse`（`safeJson` 兜底损坏数据），标量列改驼峰（`full_name`→`fullName`、`coord_lat/lng`→`coord:[lng,lat]`）返回前端。
- **为什么 JSON 列不建索引**：SQLite 不支持 JSONB 索引，且百科只有几百条数据，标签筛选用 `tags LIKE %词%` 全表扫描完全够用。
- **coord 存两列 `coord_lat/coord_lng`**：地图打点要按经纬度查，标量列比从 JSON 里取更直接。

首页 2D 地图（`ChinaMap2D.vue`）**不落库**——下钻状态（`mapDrill` 的 stack）是纯前端运行时状态，geoJSON 由 `geoLoader` 按 adcode 加载缓存，村庄散点坐标来自 villages 表的 `coord`（`coord_lng/coord_lat` 两列）。

## 8. 设计取舍问答（验收备答）

**Q：为什么档案主体用 JSON 整存，不拆关系表？**
A：档案结构随三阶段演进、嵌套深、字段多变。JSON payload 让结构演进不改表；查询/排序需要的少数字段（team_id/stage/updated_at）提成标量列 + 复合索引，兼顾灵活与查询效率。

**Q：材料文件为什么不存数据库？**
A：文件（尤其音视频）体积大，存 DB 会拖垮性能、撑大备份。存盘 + DB 存 url 是标准做法；`express.static` 只读托管 `uploads/practice`。

**Q：AI 抽取结果为什么不直接写档案？**
A：AI 可能抽错或臆造。先进不落库的 `draft`，队员逐条审校采纳才写 `collected`，保证入库数据都经人工确认——这是实训的 AI 合规底线。

**Q：collected 里的字段为什么和 AI 抽取契约一样？**
A：故意对齐，让"采纳草稿"变成纯 `push`、无字段转换，降低出错面；同时事实字段（name/quote/value）与加工字段（story/insight）分层清晰，可追溯哪些是原文、哪些是 AI 归纳。
