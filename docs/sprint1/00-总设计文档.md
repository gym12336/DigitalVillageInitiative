## 一、一句话讲清这个项目

**数乡计划**是一个连接乡村与大学生实践的数字平台。核心是一条主线：

> 大学生组队 → 选题定方案（**实践前**）→ 下乡采集材料、AI 结构化（**实践中**）→ 把成果做成可视化作品（**实践后**）。

技术栈一句话：**Vue3 + Vite 前端，Express + SQLite 后端，DeepSeek 做 AI，全程 AI 辅助但人工审校决策。**

## 二、整体架构（一张图看懂）

```
前端 Vue3                        后端 Express                外部
┌─────────────┐   HTTP/JSON    ┌──────────────────┐       ┌──────────┐
│ 首页地图     │ ─────────────> │ Route 路由层      │       │ DeepSeek │
│ 乡村百科     │                │  ↓ 薄胶水         │ ────> │  (AI)    │
│ 我的实践     │ <───────────── │ Service 服务层    │       └──────────┘
│  三阶段工作台 │    JWT 鉴权     │  ↓ 业务+授权+AI   │       ┌──────────┐
└─────────────┘                │ Lib 工具 / DB     │ ────> │ SQLite   │
                               └──────────────────┘       │ (单文件) │
                                                           └──────────┘
```

**后端严格四层，依赖自上而下：**

| 层 | 干什么 | 铁律 |
|----|--------|------|
| Route | 解析参数 → 调 service → 返回响应 | 不写业务、不写 SQL |
| Service | 业务逻辑 + 授权判定 + 编排 AI | 注入 db，不碰 HTTP |
| Lib | 纯函数/薄封装（JWT、AI 客户端、限流） | 不依赖 DB，可单测 |
| DB | 建表、迁移、种子数据 | 纯 SQL |

**三条设计原则**（贯穿全项目，被问必答）：
1. **JSON payload 存主体**：档案/作品结构多变，关键字段提列建索引，主体整存 JSON 文本，改结构不改表。
2. **AI 辅助、人决策**：AI 产出永远先进"待审校区"，人工采纳才入库，绝不自动覆盖人写的内容。
3. **AI 失败不阻断**：AI 是外部依赖会挂，所有 AI 接口失败也返回结果（带 `source` 标记降级），不让用户卡住。

## 三、【重点】实践中 —— AI 采集工作台（我负责）

**要解决的问题**：队员下乡采集的是一堆非结构化材料（访谈、文档、表格、照片、音视频），没法直接变成成果。

**这条管线把「一堆文件」变成「档案里的结构化词条」：**

```
上传材料 → 文档/表格自动解析出纯文本 → AI 抽取（人物/指标/材料要点）
  → 队员逐条审校采纳 → 回填档案 → 四维进度看板对照方案督进
```

**四个讲解要点：**

1. **按类型分档上传**（`UploadPanel.vue` + `mediaService.js`）
   音视频 200MB / 图片 20MB / 文档表格 10MB。文档表格自动解析文本，图片音视频只存档。存盘时**重铸文件名**（防路径穿越），按档案分目录。

2. **文本抽取**（`lib/fileText.js`）
   按扩展名分派：docx→mammoth，pdf→pdf-parse，xlsx→xlsx，csv→papaparse。解析库依赖注入，便于单测。超 2 万字截断防撑爆 token。

3. **AI 结构化提取 + 三级兜底**（`practiceExtractService.js`）⭐ **这是可手写讲解的 AI 禁飞区核心**
   调 DeepSeek 抽出人物/指标/材料，每条带 `confidence` 和 `source:'auto'`。
   **恒返回 200**：空输入→`source:'empty'`，AI 失败→`source:'error'`+空结果，成功→`source:'ai'`。前端据此降级不崩溃。

4. **审校回填 + 四维督进**（`StageTrack.vue` + `gapAnalysis.js`）
   AI 抽取先进 `draft` 待审校区（右栏三 Tab：指标/材料/人物），队员逐条「采纳/丢弃」，采纳才写档案。
   **动态督进**：对照 `startDate/endDate` 算时段进度，对照任务/指标/材料/人物四维算完成度，**时段过半但完成度不足**就 warn 提醒。

**加分项**：ZIP 整包导入（`zipImportService.js`）—— 解压后逐文件复用上面整条管线，防 zip 炸弹（≤300MB/≤100 文件），单文件失败不阻断整体。

> 详见 [01-架构 §1-9](01-架构设计文档.md)、[02-接口 §1-4](02-接口设计文档.md)、[03-数据模型 §1-6](03-数据模型设计文档.md)。

## 四、【重点】实践后 —— 低代码可视化搭建台（队友负责）

**要解决的问题**：实践中采集好的结构化数据，怎么变成可展示、可导出的成果作品。

**两级工作台 + 数据插槽：**

```
StageResult.vue（成果页）
  ├─ ResultCards：用档案 collected 数据自动生成成果卡（人物墙/指标/材料）
  └─ 「进入搭建台 DIY」→ /builder?dossierId=xxx
        │
   低代码可视化搭建台
    ① 编辑台画布（editor）：拖拽组件（图表/数据表/时间线），组件从 dossier 数据取数
    ② 展示工作台（display）：把大组件排进大屏布局，出成果大屏
    ③ 大组件模板（big-component）：可复用的组件组
        │
    产出 → works 作品（team 共享，可导出）
```

**三个讲解要点：**

1. **数据来源打通**：搭建台的组件通过 `dataBinding` 从**实践中采集的 dossier** 取数——这就是"实践中→实践后"的数据流闭环。
2. **画布存 JSON**：`builder_documents` 表按 `type`（editor/display/big-component）存画布 payload（组件列表 + 栅格布局 + 主题），editor/display 每档案各一条（upsert）。
3. **作品独立成表**：`works` 表与 dossiers 同构（team_id 归属、payload 整存），`source_dossier` 记录数据源档案。

> 详见 [04-架构 §4.4](04-架构设计文档(全平台完整版).md)、[05-接口 §5,§8](05-接口设计文档(全平台完整版).md)、[06-数据模型 §3.5,§3.6,§5.2,§5.3](06-数据模型设计文档(全平台完整版).md)。

## 五、支撑板块（简述，被问再展开）

- **首页 2D 下钻地图**（我负责）：echarts geo 地图，全国→省→市→区县四级下钻。下钻逻辑用纯函数状态机 `mapDrill.js`（不可变，可单测），与渲染解耦；右上角搜索面板搜村庄/地区，命中村庄放大定位。详见 [01-架构 §10](01-架构设计文档.md)。
- **乡村百科**（我负责）：村庄数据库，列表检索（分页+关键词+省份+标签）、详情页、标签词频聚合、AI 内容增强（人工优先不覆盖）。详见 [01-架构 §11](01-架构设计文档.md)。
- **实践前**（队友）：idea 输入 → 平台资源检索 + 联网搜索（博查）→ AI 分阶段方案生成（规则兜底）。
- **认证与队伍**（队友）：JWT + bcrypt；用户↔队伍多对多（memberships），档案/作品按 team 共享。
- **乡村之声、乡村好物**：需求发布与展示（本期只读 + 计数落库）。

## 六、数据模型速记（一张表记住）

| 表 | 存什么 | 关键设计 |
|----|--------|---------|
| `users` | 用户 | 密码 bcrypt 哈希，不存明文 |
| `teams` / `memberships` | 队伍 / 成员关系 | 用户↔队伍多对多，邀请码入队 |
| `dossiers` | 实践档案 ⭐ | **实践中采集数据存在 payload.collected**；team 共享 |
| `builder_documents` | 搭建台画布 | 按 type 分 editor/display/big-component |
| `works` | 成果作品 ⭐ | 实践后产出，source_dossier 关联数据源 |
| `villages` | 村庄百科 | 标量列（可查/排序）+ JSON 列（富内容） |
| `demands` / `qa` | 乡村之声 | 本期只读 + 计数 |

**共同模式**：能查/排序的字段提成标量列建索引，复杂内容存 JSON 文本列。SQLite 单文件、零部署，几百条数据全表扫描够用；预留了迁移 PostgreSQL 的路径。
