# 数乡计划 · 可视化展示形式（mark）清单

- 日期：2026-07-06
- 用途：`vizgrammar/schema.js` 的原子零件（mark）字典 + `compile.js` 的「意图 → mark」选择依据
- 关联：[成果可视化生成工具设计](2026-07-06-practice-result-visualizer-design.md) 第 4 节 Spec 契约

> 展示形式不按「图表种类」堆，而按**队伍想讲什么（叙事意图）**归类——这对齐设计文档「意图驱动」的内核。
> compile 编译器（禁飞区①）的工作，本质就是：读自然语言意图 + 数据形状 → 从本表挑 mark、拼 encoding。

## 一、讲「变化」——帮扶前后对比（最核心）

三下乡成果的灵魂是「这支队带来了什么改变」，对应 `before_after` 数据。

| mark | 形态 | 吃的数据 | 适用 |
|------|------|----------|------|
| `dumbbell` | 哑铃图：一线连前值·后值两点 | `before_after` | 多指标同看「从哪到哪」（文档已举例） |
| `bar-compare` | 前后并列柱：灰柱(前)+彩柱(后) | `before_after` | 指标少、看绝对量差 |
| `slope` | 斜率图：前后两列点连斜线 | `before_after` | 强调涨跌方向与幅度 |
| `delta-badge` | 涨跌徽标：+58% ▲ | `metrics[].delta` | 嵌 KPI 卡的单点强调 |

## 落地建议

### 一期主力（别全上）

schema 先落 **3~4 种**即可撑起最小闭环——每加一个 mark 都要在 `renderer.js` 写分支 + 补单测，是实打实的成本；文档第 8 节的可测性优势，只有在 mark 数量克制时才守得住。

推荐一期集合：

1. `dumbbell` —— 讲变化（帮扶前后）
2. `stat-row` + `radar`（或 `map-outline`+`facet-callout` 发散图）—— 讲概览
3. `timeline` —— 讲过程
4. `people-card` —— 讲人

### 二期扩展

`geo-scatter` / `flow-map` / `heat-region` 等地图类，随第二阶段 Generative UI 纵深一起上。

### DIY 场景：区分「AI 生成」与「手填」

搭建台面板需要知道每个 mark 是 AI 从数据里编排出来的，还是队员手动拖入直接填内容的。建议 schema 给每个 mark 加一个 `authoring` 标记：

- `ai` —— compile 从数据编排，如 `radar` / `dumbbell` / `sankey` / `heat-region`
- `manual` —— 队员在搭建台手填内容块，如 `title-band` / `quote-block` / `video-embed` / `cover-hero`
- `both` —— 两种来源皆可，如 `stat-row` / `people-card` / `data-table`

DIY 面板据此决定：哪些块能空手拖出来填、哪些必须先有数据才生成。

### DIY 面板：别让选择爆炸

一次全铺 20+ 种 mark，队员反而不会选。按上面**叙事意图**分组做引导式选择（「想讲变化 / 想讲人 / 想讲过程」），既降低选择成本，又复用 compile 的意图分类。

### 与三个核心的关系

- **schema.js**：本表的 mark 名 + 各自合法 encoding 键，构成「约束下自由组合」的**约束边界**。
- **compile.js（禁飞区①）**：以本表「意图 → mark」映射为选择依据，把 NL + 数据编译成 Spec。
- **renderer.js（禁飞区②）**：每个 mark 一个渲染分支，纯函数、可单测。
